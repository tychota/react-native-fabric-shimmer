## What it does

A React Native component that shows a shimmer loading state for any component
without requiring that component to be modified. The real component renders
with mock data while `loading=true`; the library measures its native layout
via Fabric's synchronous JSI `measureLayout`, and paints shimmer rectangles
at the exact positions.

- Works on **React Native ≥ 0.76** with the New Architecture (Fabric).
- Requires **Reanimated ≥ 3.17** and **React ≥ 19**.
- **React Compiler safe** — no manual `useMemo` or `React.memo` required.
- **Accessible by default** — `progressbar` role, content hidden from VoiceOver while loading, reduce-motion respected.

## Quick look

import { Skeleton } from "react-native-dynamic-shimmer";

</Skeleton>;
```

## The loading contract

Consumers decide what to render while `loading=true`. The library does not generate mock data. Typical patterns:

### Nullable data + mock fallback

```tsx
<Skeleton loading={!user} baseColor={colors.base} highlightColor={colors.highlight}>
  <UserCard user={user ?? MOCK_USER} />
</Skeleton>
```

### Stale-while-revalidate

```tsx
const { data, isValidating } = useSWR('/user/1', fetchUser)

<Skeleton loading={isValidating && !data} baseColor={colors.base} highlightColor={colors.highlight}>
  <UserCard user={data ?? lastKnownUser} />
</Skeleton>
```

### Keep shimmer showing even with data

```tsx
// Show skeleton while a background fetch is in flight, regardless of data presence.
<Skeleton loading={isRefetching} baseColor={colors.base} highlightColor={colors.highlight}>
  <UserCard user={user} />
</Skeleton>
```

## Anti-flicker behavior

By default:

- If `loading` resolves within roughly one React render cycle, **the skeleton does not appear**. No render, no measurement, no animation.
- Once the skeleton is visible, it stays visible for at least `minShowDuration` (default **500 ms**) before transitioning off.
- `transition` (default **300 ms**) controls the fade-out.

These defaults prevent flicker on fast cache hits and smooth the transition out.

## Error states

The library doesn't prescribe an error UI. Render your own when the query fails and let the skeleton handle the "still loading" path:

```tsx
if (error) return <ErrorCard error={error} onRetry={refetch} />;
return (
  <Skeleton loading={!data} baseColor={colors.base} highlightColor={colors.highlight}>
    <UserCard user={data ?? MOCK_USER} />
  </Skeleton>
);
```

## Lists

Wrap one `<Skeleton>` per row, not one around the whole list. Pass `null` slots during loading and render the card with mock data:

```tsx
<FlashList
  data={users ?? new Array(20).fill(null)}
  renderItem={({ item }) => (
    <Skeleton loading={item === null} baseColor={colors.base} highlightColor={colors.highlight}>
      <UserCard user={item ?? MOCK_USER} />
    </Skeleton>
  )}
/>
```

See also: [List rows recipe](/recipes/list-rows/).

## `<Skeleton>`

```ts
type SkeletonProps = {
  // Required
  loading: boolean;
  children: ReactNode;
  baseColor: string;
  highlightColor: string;

  // Optional — visual
  animation?: "shimmer" | "pulse" | "none"; // default 'shimmer'
  transition?: boolean | number; // ms; default 300, false disables
  minShowDuration?: number; // ms; default 500

  // Optional — extension hooks
  renderBone?: RenderBoneFn;
  classify?: ClassifyFn;
  refineBones?: RefineBonesFn;

  // Optional — observability
  onMeasured?: (bones: ReadonlyArray<BoneRect>) => void;

  // Optional — layout / a11y
  style?: StyleProp<ViewStyle>;
  accessibilityLabel?: string; // default "Loading"
};
```

### Required props

- **`loading`** — when `true`, render the skeleton; when `false`, render children as-is. Transition and sticky duration are handled internally.
- **`children`** — the real component. Pass valid data (real or mock) so layout can be measured. Children always render while loading; they're hidden via `opacity: 0`.
- **`baseColor`** — background color of each bone.
- **`highlightColor`** — sweep color for shimmer, pulse target for pulse.

### Optional visual props

- **`animation`** — `'shimmer'` (default) is a translating gradient sweep; `'pulse'` is an opacity oscillation; `'none'` renders static bones (useful for test snapshots).
- **`transition`** — milliseconds for the fade-out on `loading=false`. `true` uses 300ms; `false` disables the fade entirely.
- **`minShowDuration`** — milliseconds. Once the skeleton appears, it stays for at least this long to prevent flicker.

### Extension hooks

- **`renderBone`** — replaces the default `<Bone>` per bone. See [renderBone](/extension/render-bone/).
- **`classify`** — overrides the default fiber classifier. See [classify](/extension/classify/).
- **`refineBones`** — transforms the IR tree before flattening. See [refineBones](/extension/refine-bones/).

### Observability

- **`onMeasured`** — called with the flattened `BoneRect[]` each time measurement completes. Useful for debug overlays, analytics, or memoization.

## `<Bone>`

```ts
type BoneProps = {
  rect: BoneRect;
  ctx: BoneContext;
};
```

The default bone renderer. Exposed so custom `renderBone` implementations can
delegate the default visual back in for bones they don't want to override.

## Types

```ts
type BoneKind = "text" | "image" | "view" | "container";

type BoneRect = {
  readonly x: number;
  readonly y: number;
  readonly width: number;
  readonly height: number;
  readonly borderRadius: number | "50%";
  readonly kind: BoneKind;
  readonly backgroundColor?: string;
  readonly borderColor?: string;
  readonly borderWidth?: number;
};

type BoneContext = {
  readonly progress: SharedValue<number>;
  readonly baseColor: string;
  readonly highlightColor: string;
  readonly animation: "shimmer" | "pulse" | "none";
  readonly index: number;
  readonly total: number;
};

type BoneNode = {
  readonly id: string;
  readonly type: string;
  readonly classification: "leaf" | "container" | "transparent";
  readonly rect: {
    readonly x: number;
    readonly y: number;
    readonly width: number;
    readonly height: number;
  };
  readonly style: {
    backgroundColor?: string;
    borderRadius?:
      | number
      | { topLeft?: number; topRight?: number; bottomLeft?: number; bottomRight?: number };
    borderColor?: string;
    borderWidth?: number;
  };
  readonly children: ReadonlyArray<BoneNode>;
};

type FiberNode = {
  readonly type: string | Function | null;
  readonly memoizedProps: unknown;
  readonly stateNode: unknown;
  readonly child: FiberNode | null;
  readonly sibling: FiberNode | null;
  readonly return: FiberNode | null;
};

type FiberClassification = "leaf" | "container" | "transparent" | "skip";

type RenderBoneFn = (rect: BoneRect, ctx: BoneContext) => ReactNode;
type ClassifyFn = (fiber: FiberNode) => FiberClassification;
type RefineBonesFn = (tree: BoneNode) => BoneNode;
```

## IR helpers

Exported for use inside `refineBones`:

- `walk(tree, visitor)` — depth-first pre-order traversal; visitor receives `(node, path)`.
- `find(tree, predicate): BoneNode | null` — first match.
- `findAll(tree, predicate): BoneNode[]` — all matches, in pre-order.
- `hide(tree, target): BoneNode` — returns a new tree with the target subtree removed.
- `merge(tree, targets: BoneNode[]): BoneNode` — returns a new tree with the targets replaced by one bone at their union rect.
- `union(rects): Rect | null` — bounding rect utility.

## Default classifier

Exported so custom `classify` functions can delegate for nodes they don't want to override:

```ts
import { defaultClassify, type ClassifyFn } from "react-native-dynamic-shimmer";

const classify: ClassifyFn = (fiber) => {
  const t = fiber.type;
  if (typeof t === "function" && (t as { displayName?: string }).displayName === "MyIcon")
    return "leaf";
  return defaultClassify(fiber);
};
```

- [ ] **Step 1: `theory/yoga.mdx`**

```mdx
---
title: Yoga layout
description: Why measuring the rendered component captures exactly the right rectangles.
---

React Native uses Yoga for flex layout. Yoga computes positions and sizes from
the `style` props on host views — `flexDirection`, `flex`, `padding`, `margin`,
`width`, etc. This computation is done in native C++ after every React commit
that changes the layout tree.

By the time `onLayout` fires, Yoga has already resolved every rect. The shadow
tree holds authoritative positions, and our library reads them via Fabric's
`measureLayout`.

## Why we render children before measuring

A common question: "can we compute bone positions from the component's source
code, without rendering it?" No — Yoga needs an actual tree to lay out. The
container's size (which depends on ancestors, font scaling, OS-level Dynamic
Type, locale, RTL) is only known at runtime.

Rendering with mock data, measuring, then painting shimmer is the smallest
thing that captures the full runtime layout.

## Text wrapping

`<Text>` uses native text measurement, which depends on font, font scale,
locale, and the rendered container width. We can't predict where lines will
wrap from source. The runtime measurement catches the actual line breaks —
each rendered `<Text>` node reports its own rect, even across multiple lines
(each line is a separate host text run on most platforms).

## Constraint kinks

Yoga layout is **piecewise linear** in container width: within any single
constraint region, positions are `a*W + b`. Constraints (`minWidth`,
`maxWidth`, `flexWrap`, intrinsic content size, `aspectRatio`) introduce kinks
where the slope changes. Our runtime measurement doesn't need to reason about
these — we just observe the result at the actual container width.
```

- [ ] **Step 2: `theory/fabric-jsi.mdx`**

````mdx
---
title: Fabric & JSI measurement
description: How measureLayout became a synchronous C++ call and why it matters.
---

On the old architecture (Paper), `measureLayout` went through the async bridge:

```
JS thread → bridge → UI thread (ObjC/Java) → UIManager → callback → bridge → JS
```

Every measurement cost a bridge round-trip — sub-millisecond but not free, and
not composable with synchronous work. Measuring 30 leaves on Paper meant 30
round-trips.

On the **New Architecture** (Fabric), `measureLayout` is a direct JSI call
into C++:

```
JS thread → JSI → C++ ShadowNode::getRelativeLayoutMetrics → return
```

The callback-based API is a legacy shape; the underlying call is synchronous.
A single measurement is microseconds. Thirty measurements in parallel complete
well under 1 ms on a modern device.

## What this means for skeleton libraries

The original "dynamic shimmer skeletons" pattern, born on the web, was written
under the assumption that leaving a complex component mounted just to measure
it is expensive. On Fabric that assumption inverts: the measurement is so
cheap that there's no reason to pre-compute bones at build time.

`react-native-dynamic-shimmer` measures every time you load. The library is
simpler, always accurate, and never drifts from the code.

## Why new arch is a hard requirement

The `ReactFabricHostComponent` type on new-arch exposes `measureLayout`
directly. On old arch, `UIManager.measureLayout(childTag, parentTag, ...)` is
the API — different shape, different performance profile. Rather than ship
two code paths, we require new arch and throw a clear dev error if it's
missing.
````

- [ ] **Step 3: `theory/measurement.mdx`**

````mdx
---
title: Measurement flow
description: From onLayout to first shimmer frame, step by step.
---

The pipeline, from the moment `loading=true` is committed:

```
Frame T   : loading=true committed
Frame T   : onLayout(container) fires
            handleLayout schedules requestAnimationFrame
Frame T+1 : getFiber(ref.current) → rootFiber
            buildBoneTree(rootFiber, classify) → UnmeasuredBoneTree
            await Promise.all(leaves.map(measureLayout)) → rects attached
            refineBones(tree) → RefinedBoneTree
            flattenTree(tree) → BoneRect[]
            setState(bones)
Frame T+2 : React commits, overlay mounts, useAnimatedStyle installs
            first shimmer frame visible
```

End-to-end: typically **~50 ms** on a modern device, **~100 ms** under load.

## What each stage does

**`getFiber`** — dereferences `ref.current.__internalInstanceHandle` (the
Fabric-provided access point to the React fiber).

**`buildBoneTree`** — walks the fiber tree depth-first, calling `classify` for
each node. Emits an intermediate representation (IR) with placeholder rects.

**`measureTree`** — issues `measureLayout` in parallel for every classifiable
node, relative to the container. Drops any node whose measurement fails.

**`refineBones`** — optional user hook. Receives the IR with real rects and
returns a transformed IR. Default is identity.

**`flattenTree`** — walks the IR depth-first and emits a `BoneRect[]`,
containers before their children (so render order paints containers
behind content).

## Re-measurement

`onLayout` re-fires on orientation change, split-view resize, font-scale
change, and any layout shift. The pipeline re-runs from scratch, producing
fresh bones. The shimmer oscillator stays put (same `SharedValue`), so the
visual stays smooth during re-measurements.

## Race handling

Concurrent layout events are debounced via a monotonic run id — only the most
recent run's result applies. If the container unmounts mid-measurement, the
result is discarded.
````

## Progress-bar role

While loading is visible, the overlay has:

```ts
accessibilityRole="progressbar"
accessibilityState={{ busy: true }}
accessibilityLabel="Loading"    // overridable via the accessibilityLabel prop
```

VoiceOver / TalkBack announces "Loading, busy", then moves on. Swipe
navigation treats the overlay as one focusable region.

## Children hidden from AX

During loading, the children wrapper has
`importantForAccessibility="no-hide-descendants"` and
`accessibilityElementsHidden={true}`. Mock data is never announced. Once
`isVisible` flips to false and the fade-out completes, these flags flip back
and the real content becomes focusable.

## Reduce Motion

Reanimated's `ReduceMotion.System` enum is passed to every `withTiming` and
`withRepeat` in the library. When the OS-level "Reduce Motion" setting is
enabled, animations short-circuit to their target value — the shimmer gradient
sits at its middle position, unmoving. It's still recognizable as a loading
indicator.

No JS branching needed: consumers don't have to detect the setting themselves.

## Dynamic Type / font scaling

The library measures at runtime. When a user increases system font size, the
rendered children (which we measure) are already at the scaled size, and the
bones match automatically.

## High contrast

Colors are flat props (`baseColor`, `highlightColor`). Consumers pick values
that meet their contrast requirements. We don't prescribe — your design
system's tokens apply.

- [ ] **Step 5: `theory/react-compiler.mdx`**

```mdx
---
title: React Compiler safety
description: How the library stays compiler-clean without escape hatches.
---

React 19 ships the React Compiler as a stable optimization. The library is
designed to work cleanly both with and without the compiler — no
`'use no memo'` directives, no surprising behavior.

## What we avoid

- **No manual `React.memo` / `useMemo` / `useCallback`.** The compiler adds
  memoization where it's safe; we don't duplicate that.
- **No render-time ref reads.** Fiber walking and `measureLayout` run inside
  `onLayout` handlers and `requestAnimationFrame`, not during render.
- **No mutation of props or state during render.** Measurement results land
  via `setState`, which the compiler handles correctly.

## Reanimated interop

Reanimated worklets are opaque to the React Compiler because `'worklet'` is a
separate compilation path. The compiler leaves worklet functions alone.
`useAnimatedStyle(() => { 'worklet' ... })` is safe under the compiler — the
callback's dependencies are the worklet's closure, not React hook deps.

## Users writing extension hooks

The library's extension hooks (`classify`, `refineBones`, `renderBone`) are
**pure functions** called during the measurement pipeline, outside React's
render phase. The compiler doesn't see them as dependencies; users can pass
inline functions without penalty.

If a custom component passed via `renderBone` uses React hooks, those hooks
are subject to the compiler normally — same rules as any other component.

- [ ] **Step 4: `extension/classify.mdx`**

````mdx
---
title: classify
description: Control which fibers become bones.
---

```ts
type FiberClassification = "leaf" | "container" | "transparent" | "skip";
type ClassifyFn = (fiber: FiberNode) => FiberClassification;
```

- `'leaf'` — emit a bone, **stop descending** into children.
- `'container'` — emit a bone **and continue descending** into children.
- `'transparent'` — don't emit, descend (plain positioning wrapper).
- `'skip'` — don't emit, don't descend (hidden subtree).

## Default

The library ships a 4-way classifier:

| Fiber type                                                       | Classification |
| ---------------------------------------------------------------- | -------------- |
| `Text` / `RCTText` / `TextInput`                                 | `leaf`         |
| `Image` / `RCTImage`                                             | `leaf`         |
| View with `display: 'none'` or `opacity: 0`                      | `skip`         |
| Childless View with background, border, or radius                | `leaf`         |
| View with children **AND** background / border / radius / shadow | `container`    |
| Plain View wrapper                                               | `transparent`  |

## Delegating to default

For custom components that should be treated as one bone, override just those:

```tsx
import { Skeleton, defaultClassify, type ClassifyFn } from 'react-native-dynamic-shimmer'

const classify: ClassifyFn = (fiber) => {
  const t = fiber.type
  if (typeof t === 'function' && (t as { displayName?: string }).displayName === 'MyIconBadge') {
    return 'leaf'  // treat the whole badge as one bone
  }
  return defaultClassify(fiber)
}

<Skeleton loading classify={classify} ...>
```

## Skipping subtrees

```tsx
const classify: ClassifyFn = (fiber) => {
  const t = fiber.type;
  if (typeof t === "function" && (t as { displayName?: string }).displayName === "DebugOverlay") {
    return "skip"; // don't shimmer the debug overlay
  }
  return defaultClassify(fiber);
};
```

````mdx
---
title: refineBones
description: Transform the bone tree before flattening — hide, merge, reshape.
---

```ts
type RefineBonesFn = (tree: BoneNode) => BoneNode;
```

`refineBones` receives the measured IR — a tree of plain JS objects — and
returns a (new) tree. It runs after measurement, before flattening.

## The tree

```ts
type BoneNode = {
  readonly id: string
  readonly type: string         // 'View', 'RCTText', 'UserCard', etc.
  readonly classification: 'leaf' | 'container' | 'transparent'
  readonly rect: { x: number; y: number; width: number; height: number }
  readonly style: { backgroundColor?: string; borderRadius?: number | {...}; ... }
  readonly children: ReadonlyArray<BoneNode>
}
```

No React internals. Just data you can traverse and transform.

## Helpers

```ts
import { walk, find, findAll, hide, merge, union } from "react-native-dynamic-shimmer";
```

- **`walk(tree, visit)`** — depth-first pre-order traversal.
- **`find(tree, predicate)`** — first match, or `null`.
- **`findAll(tree, predicate)`** — all matches, in pre-order.
- **`hide(tree, target)`** — returns a new tree with the target subtree
  removed.
- **`merge(tree, targets[])`** — replaces the targets with one bone at their
  union rect. Useful for tight line spacing.
- **`union(rects)`** — bounding rect utility.

## Patterns

### Hide a decorative icon

```tsx
refineBones={(tree) => {
  const chevron = find(tree, n => n.type === 'Chevron')
  return chevron ? hide(tree, chevron) : tree
}}
```

### Merge adjacent text lines

```tsx
refineBones={(tree) => {
  const texts = findAll(tree, n => n.type === 'RCTText' && n.rect.y < 40)
  return texts.length >= 2 ? merge(tree, texts.slice(0, 2)) : tree
}}
```

### Enlarge a specific bone

```tsx
refineBones={(tree) => {
  // Not provided by the library — build your own with walk + immutability.
  // (Skipped for brevity — see the example app's refine-bones-merge screen
  //  for a complete pattern.)
  return tree
}}
```

## When to use

- `classify` for **structural** decisions (what counts as a leaf).
- `refineBones` for **semantic** decisions (these two bones should merge; this
  bone should go away).
- `renderBone` for **visual** decisions (this bone should look different).

Each hook is at a different layer; they compose without surprising each
other.

description: Dev warnings, common gotchas, and how to resolve them.

## Warnings

The library logs at most five distinct warnings in dev, each at most once per
Skeleton mount.

### `Fabric required. Enable the New Architecture.`

You're on the old architecture (Paper). The library doesn't support it.

Fix: enable new arch.

- **Expo:** set `newArchEnabled: true` in `app.config.ts` and rebuild.
- **Bare RN:** set `newArchEnabled=true` in `android/gradle.properties` and
  run `RCT_NEW_ARCH_ENABLED=1 pod install --repo-update` in `ios/`.

### `Container ref has no fiber instance. Is this inside a <Modal> or <Portal>?`

The `ref.current` doesn't expose `__internalInstanceHandle`. Common causes:

- The Skeleton is rendered inside a `<Modal>` or `<Portal>` whose children
  live in a detached tree.
- The container is optimized away by Fabric view flattening.

Fix: wrap the Skeleton (or the Modal) with `<View collapsable={false}>` or
rerender outside the portal.

### `No bones found after walk. Your subtree may contain only transparent views.`

The measurement completed but no leaves were classified as bone candidates.
The subtree may be only plain `<View>` wrappers without text, images, or
distinctive surfaces.

Fix: add a leaf type (`<Text>`, `<Image>`) to the children, or provide a
`classify` function that marks your custom component as a leaf.

### `{hookName} threw: {error}. Falling back to default.`

Your `classify` / `refineBones` / `renderBone` threw. The library falls back
to the default.

Fix: handle the error in your hook. Check types and nullish guards.

### `Measurement pipeline exceeded {N}ms for {M} nodes.`

The pipeline took longer than expected, usually because the subtree is very
large (50+ leaves). Skeleton rendering still works, but consider narrowing
the scope with `classify`:

```tsx
<Skeleton classify={(fiber) => fiber.type === 'DebugBorder' ? 'skip' : defaultClassify(fiber)}>
```

## Common issues

### Bones don't appear even when `loading=true`

Check:

1. Is `loading=true` actually true? Log it.
2. Did `onLayout` fire? Add a temporary `onMeasured` prop to verify.
3. Is the container ref attached? (see "Container ref has no fiber instance".)
4. Are the children rendering with valid data? A subtree with all transparent
   `<View>` wrappers has no measurable leaves.

### Shimmer looks choppy

Usually a Reanimated or new-arch config issue:

- Ensure `react-native-worklets/plugin` is **last** in `babel.config.js`.
- Ensure new arch is enabled.
- Profile with the Reanimated debugger to confirm worklets run on the UI thread.

### Visual differs between iOS and Android

Fabric's view flattening and measurement rounding may differ slightly. The
library measures whatever Fabric reports; minor pixel differences are
platform-native. If they're severe, open an issue with a repro.

### React Compiler complains about my hook

Ensure:

- Your component doesn't mutate props or state during render.
- You're not reading a mutable ref during render (use in effects or handlers).
- You're not calling React hooks inside `classify` / `refineBones` /
  `renderBone` — those are pure functions called outside render.
````
